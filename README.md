I developed this project to explore the relationship between object oriented data management and interactive game loops. The goal was to build a terminal based RPG where players could create custom characters, distribute skill points into specific stats, and engage in scaled battles against an AI controlled enemy. I focused heavily on the math behind the combat, ensuring that defense, armor, and critical hit chances all factor into a balanced damage calculation system.

The technical foundation of the game is the data class, which I used to blueprint the player's attributes like health, damage, and defense. Instead of using loose variables, I used this class to instantiate player objects and store them in a global list. This allowed me to easily implement features like searching for a player by name, displaying full stat sheets, or deleting players from the party. It was a great way to practice encapsulation and learn how to manage a dynamic "character roster" within a running application.

The combat engine uses a loop based system where players and enemies take turns attacking. I implemented a "Skill Point" (SP) distribution system during character creation, allowing for different playstyles such as a high health "Tank" or a high damage "Glass Cannon." During battle, the damage logic includes a randomized critical hit multiplier and a defense reduction formula. To keep the game challenging, I added a difficulty scaling variable that increases every time an enemy is defeated, ensuring that the randomly generated monsters grow stronger alongside the player.

Since this is a terminal game, I used the os and time modules to create a cleaner user experience. By clearing the screen between turns and adding artificial delays with time.sleep(), I was able to simulate the "pacing" of a real battle, making the text output feel less overwhelming. I also included input validation for player names and menu choices to prevent the game from crashing if the user enters an unexpected value. This project taught me how to bridge the gap between back end data logic and a front end user interface that responds to player decisions.
